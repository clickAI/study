# Part 3: 지연 로드 도구 (번역)

> ToolSearch로 가져온 Deferred 도구들의 한국어 번역입니다.

## AskUserQuestion (사용자에게 질문하기)

```
설명: 실행 중에 사용자에게 질문해야 할 때 이 도구를 사용하세요. 다음을 가능하게 합니다:
1. 사용자 선호도나 요구사항 수집
2. 모호한 지시 명확화
3. 작업 중 구현 선택에 대한 결정 얻기
4. 사용자에게 방향에 대한 선택지 제공

사용 참고:
- 사용자는 항상 "기타"를 선택하여 사용자 정의 텍스트를 입력할 수 있습니다
- multiSelect: true를 사용하여 질문에 대해 여러 답변을 선택할 수 있게 합니다
- 특정 옵션을 추천하면, 그 옵션을 목록의 첫 번째로 만들고 레이블 끝에 "(권장)"을 추가하세요

계획 모드 참고: 계획 모드에서는 이 도구를 사용하여 계획을 확정하기 전에 요구사항을 명확히 하거나 접근 방식 사이에서 선택하세요. "계획이 준비되었나요?" 또는 "진행해야 하나요?"를 묻기 위해 이 도구를 사용하지 마세요 - 계획 승인에는 ExitPlanMode를 사용하세요. 중요: 질문에서 "계획"을 참조하지 마세요 (예: "계획에 대한 피드백이 있나요?", "계획이 괜찮아 보이나요?") ExitPlanMode를 호출할 때까지 사용자는 UI에서 계획을 볼 수 없기 때문입니다.

미리보기 기능:
사용자가 시각적으로 비교해야 하는 구체적 산출물을 제시할 때 옵션의 선택적 `preview` 필드를 사용하세요:
- UI 레이아웃이나 컴포넌트의 ASCII 목업
- 다른 구현을 보여주는 코드 스니펫
- 다이어그램 변형
- 설정 예시

매개변수:
- questions (필수, 배열, 1-4개): 사용자에게 물을 질문들
  - question (필수, 문자열): 완전한 질문
  - header (필수, 문자열): 칩/태그로 표시되는 매우 짧은 레이블 (최대 12자)
  - options (필수, 배열, 2-4개): 사용 가능한 선택지
    - label (필수, 문자열): 표시 텍스트 (1-5단어)
    - description (필수, 문자열): 선택지의 설명
    - preview (선택, 문자열): 미리보기 내용
  - multiSelect (필수, 불리언, 기본값 false): 복수 선택 허용 여부
- answers (선택, 객체): 권한 컴포넌트가 수집한 사용자 답변
- annotations (선택, 객체): 질문별 주석
- metadata (선택, 객체): 추적 및 분석 목적의 메타데이터
```

## CronCreate (크론 작업 생성)

```
설명: 미래 시점에 프롬프트를 대기열에 넣도록 예약합니다. 반복 일정과 일회성 알림 모두에 사용합니다.

사용자의 로컬 시간대에서 표준 5필드 cron을 사용합니다: 분 시 일 월 요일. "0 9 * * *"은 로컬 오전 9시를 의미합니다 — 시간대 변환이 필요 없습니다.

## 일회성 작업 (recurring: false)
"X시에 알려줘" 또는 "<시간>에 Y해줘" 요청용 — 한 번 실행 후 자동 삭제.

## 반복 작업 (recurring: true, 기본값)
"N분마다" / "매시간" / "평일 오전 9시" 요청용.

## :00과 :30 분 표시를 피하세요
모든 사용자가 "오전 9시"를 요청하면 `0 9`가 되어, 전 세계의 요청이 같은 순간에 API에 도착합니다. 사용자의 요청이 대략적이면, 0이나 30이 아닌 분을 선택하세요.

## 세션 전용
작업은 이 Claude 세션에서만 존재합니다 — 디스크에 기록되지 않으며, Claude 종료 시 작업이 사라집니다.

## 런타임 동작
작업은 REPL이 유휴 상태일 때만(쿼리 중이 아닐 때) 실행됩니다. 반복 작업은 7일 후 자동 만료됩니다.

매개변수:
- cron (필수, 문자열): 로컬 시간 기준 5필드 cron 표현식
- prompt (필수, 문자열): 각 실행 시간에 대기열에 넣을 프롬프트
- recurring (선택, 불리언): true(기본값)=삭제되거나 7일 후 자동 만료될 때까지 매 cron 일치마다 실행. false=다음 일치에서 한 번 실행 후 자동 삭제
```

매개변수:

## EnterPlanMode (계획 모드 진입)

```
설명: 비사소한 구현 작업을 시작하려 할 때 사전적으로 이 도구를 사용하세요. 코드를 작성하기 전에 접근 방식에 대한 사용자의 동의를 얻으면 낭비되는 노력을 방지하고 조율을 보장합니다.

## 이 도구를 사용할 때

단순하지 않은 한 구현 작업에는 EnterPlanMode 사용을 선호하세요. 다음 조건 중 하나라도 해당되면 사용:

1. **새 기능 구현**: 의미 있는 새 기능 추가
2. **여러 유효한 접근 방식**: 여러 방법으로 해결 가능한 작업
3. **코드 수정**: 기존 동작이나 구조에 영향을 미치는 변경
4. **아키텍처 결정**: 패턴이나 기술 선택이 필요한 작업
5. **다중 파일 변경**: 2-3개 이상의 파일을 건드릴 가능성이 높은 작업
6. **불명확한 요구사항**: 전체 범위를 이해하기 전에 탐색이 필요
7. **사용자 선호도가 중요**: 구현이 합리적으로 여러 방향으로 갈 수 있을 때

## 이 도구를 사용하지 말아야 할 때

단순한 작업에만 건너뛰세요:
- 한 줄 또는 몇 줄 수정 (오타, 명백한 버그, 작은 조정)
- 명확한 요구사항이 있는 단일 함수 추가
- 사용자가 매우 구체적이고 상세한 지시를 준 작업
- 순수 연구/탐색 작업 (대신 explore 에이전트와 Agent 도구 사용)

## 계획 모드에서 일어나는 일

1. Glob, Grep, Read 도구를 사용하여 코드베이스를 철저히 탐색
2. 기존 패턴과 아키텍처를 이해
3. 구현 접근 방식을 설계
4. 사용자에게 계획을 제시하여 승인 받기
5. 접근 방식을 명확히 해야 하면 AskUserQuestion 사용
6. 구현할 준비가 되면 ExitPlanMode로 계획 모드 종료

매개변수: 없음
```

## ExitPlanMode (계획 모드 종료)

```
설명: 계획 모드에 있고 계획 파일에 계획 작성을 완료하여 사용자 승인을 받을 준비가 되었을 때 이 도구를 사용하세요.

## 이 도구의 작동 방식
- 계획 모드 시스템 메시지에 지정된 계획 파일에 이미 계획을 작성했어야 합니다
- 이 도구는 계획 내용을 매개변수로 받지 않습니다 - 작성한 파일에서 계획을 읽습니다
- 이 도구는 단순히 계획이 완료되었고 사용자가 검토하고 승인할 준비가 되었음을 알립니다

## 이 도구를 사용할 때
중요: 코드 작성이 필요한 작업의 구현 단계를 계획할 때만 이 도구를 사용하세요. 정보를 수집하고, 파일을 검색하고, 파일을 읽거나 일반적으로 코드베이스를 이해하려는 연구 작업에는 이 도구를 사용하지 마세요.

중요: "이 계획이 괜찮나요?"나 "진행해야 하나요?"를 묻기 위해 AskUserQuestion을 사용하지 마세요 - 이 도구가 정확히 그 일을 합니다. ExitPlanMode는 본질적으로 계획에 대한 사용자 승인을 요청합니다.

매개변수:
- allowedPrompts (선택, 배열): 계획을 구현하는 데 필요한 프롬프트 기반 권한
  - tool (필수, 문자열, 열거형: "Bash"): 이 프롬프트가 적용되는 도구
  - prompt (필수, 문자열): 행동의 의미적 설명 (예: "테스트 실행", "의존성 설치")
```

## EnterWorktree (워크트리 진입)

```
설명: 사용자가 명시적으로 워크트리에서 작업하기를 요청할 때만 이 도구를 사용하세요. 이 도구는 격리된 git 워크트리를 생성하고 현재 세션을 그곳으로 전환합니다.

## 사용할 때
- 사용자가 명시적으로 "worktree"를 말할 때 (예: "워크트리 시작", "워크트리에서 작업", "워크트리 생성")

## 사용하지 말아야 할 때
- 사용자가 브랜치 생성, 전환, 다른 브랜치에서 작업을 요청 — 대신 git 명령 사용
- 사용자가 버그 수정이나 기능 작업을 요청 — 워크트리를 구체적으로 언급하지 않는 한 일반 git 워크플로 사용
- 사용자가 명시적으로 "worktree"를 언급하지 않는 한 이 도구를 절대 사용하지 마세요

## 동작
- git 저장소에서: HEAD 기반의 새 브랜치로 `.claude/worktrees/` 안에 새 git 워크트리 생성
- git 저장소 외부에서: VCS 독립적 격리를 위해 WorktreeCreate/WorktreeRemove hooks에 위임
- 세션의 작업 디렉터리를 새 워크트리로 전환

매개변수:
- name (선택, 문자열): 워크트리의 이름. 제공하지 않으면 랜덤 이름 생성
```

## ExitWorktree (워크트리 종료)

```
설명: EnterWorktree로 생성된 워크트리 세션을 종료하고 원래 작업 디렉터리로 세션을 반환합니다.

## 범위
이 도구는 이 세션에서 EnterWorktree로 생성된 워크트리에만 작동합니다. 다음에는 영향을 미치지 않습니다:
- `git worktree add`로 수동 생성한 워크트리
- 이전 세션의 워크트리
- EnterWorktree가 호출되지 않은 경우 현재 디렉터리

## 사용할 때
- 사용자가 명시적으로 "워크트리 종료", "워크트리 나가기", "돌아가기"를 요청할 때
- 사전적으로 호출하지 마세요 — 사용자가 요청할 때만

매개변수:
- action (필수, 열거형): "keep" 또는 "remove"
  - "keep" — 워크트리 디렉터리와 브랜치를 디스크에 그대로 둠
  - "remove" — 워크트리 디렉터리와 브랜치를 삭제
- discard_changes (선택, 불리언): action이 "remove"이고 워크트리에 커밋되지 않은 파일이나 머지되지 않은 커밋이 있을 때 true 필요
```

## NotebookEdit (노트북 편집)

```
설명: Jupyter 노트북(.ipynb 파일)의 특정 셀 내용을 새 소스로 완전히 교체합니다. notebook_path 매개변수는 상대 경로가 아닌 절대 경로여야 합니다. cell_number는 0부터 시작합니다. edit_mode=insert를 사용하여 cell_number로 지정된 인덱스에 새 셀을 추가합니다. edit_mode=delete를 사용하여 해당 인덱스의 셀을 삭제합니다.

매개변수:
- notebook_path (필수, 문자열): 편집할 Jupyter 노트북 파일의 절대 경로
- new_source (필수, 문자열): 셀의 새 소스
- cell_id (선택, 문자열): 편집할 셀의 ID
- cell_type (선택, 열거형): "code" 또는 "markdown"
- edit_mode (선택, 열거형): "replace", "insert", "delete" (기본값: replace)
```

## SendMessage (메시지 보내기)

```
설명: 팀에서 에이전트 팀원에게 메시지를 보내고 프로토콜 요청/응답을 처리합니다.

## 스키마
모든 호출에는 세 필드가 있습니다:
- to: 수신자 주소 (문자열, 필수)
- message: 메시지 내용 — 일반 문자열 또는 구조화된 프로토콜 객체 (필수)
- summary: UI에 표시되는 5-10단어 미리보기

## 주소 지정 (to)
세션당 하나의 팀. 멤버 이름으로 주소 지정:
- "researcher" — "researcher"라는 팀원에게 직접 메시지
- "*" — 모든 팀원에게 브로드캐스트 (자신 제외)

구조화된 프로토콜 메시지(종료, 계획 승인)는 브로드캐스트할 수 없습니다 — 특정 수신자 이름이 필요합니다.

## 일반 텍스트 메시지
특정 팀원에게 메시지 보내기.

## 브로드캐스트 (가급적 사용 자제)
경고: 브로드캐스트는 비용이 큽니다. 모든 팀원에게 별도 메시지를 보냅니다.
정당한 사용 사례: 즉각적인 팀 전체 주의가 필요한 중요 문제, 모든 팀원에게 영향을 미치는 주요 공지.

## 구조화된 프로토콜 메시지

### 종료 요청
팀원에게 정상적 종료를 요청: { "type": "shutdown_request", "reason": "이유" }

### 종료 응답
종료 요청을 받으면 승인 또는 거부로 응답해야 합니다.
승인: { "type": "shutdown_response", "request_id": "abc-123", "approve": true }
거부: { "type": "shutdown_response", "request_id": "abc-123", "approve": false, "reason": "이유" }

### 계획 승인 응답
plan_mode_required가 있는 팀원이 ExitPlanMode를 호출하면 계획 승인 요청을 보냅니다.
승인: { "type": "plan_approval_response", "request_id": "abc-123", "approve": true }
거부: { "type": "plan_approval_response", "request_id": "abc-123", "approve": false, "feedback": "피드백" }

중요 참고:
- 팀원의 메시지는 자동으로 전달됩니다. 수신함을 수동으로 확인할 필요 없습니다.
- 항상 팀원을 UUID가 아닌 이름(예: "team-lead", "researcher")으로 참조하세요.

매개변수:
- to (필수, 문자열): 수신자: 팀원 이름 또는 "*" (전체 브로드캐스트)
- message (필수): 일반 텍스트 문자열 또는 구조화된 프로토콜 객체
- summary (선택, 문자열): 5-10단어 미리보기
```

## TaskCreate (작업 생성)

```
설명: 현재 코딩 세션을 위한 구조화된 작업 목록을 생성합니다. 진행 상황을 추적하고, 복잡한 작업을 구성하고, 사용자에게 철저함을 보여주는 데 도움이 됩니다.

## 사용할 때
- 복잡한 다단계 작업 - 3개 이상의 단계나 행동이 필요한 작업
- 비사소하고 복잡한 작업 - 신중한 계획이나 여러 작업이 필요한 작업
- 계획 모드 - 계획 모드 사용 시 작업 목록을 생성하여 작업 추적
- 사용자가 명시적으로 할 일 목록을 요청
- 사용자가 여러 작업을 제공 (번호 매기기 또는 쉼표 구분)

## 사용하지 말아야 할 때
- 단 하나의 간단한 작업만 있을 때
- 작업이 사소하고 추적이 조직적 이점을 제공하지 않을 때
- 3개 미만의 사소한 단계로 완료 가능할 때

매개변수:
- subject (필수, 문자열): 작업의 간단한 제목 (명령형)
- description (필수, 문자열): 무엇을 해야 하는지의 상세 설명
- activeForm (선택, 문자열): in_progress 시 스피너에 표시될 현재진행형 (예: "테스트 실행 중")
- metadata (선택, 객체): 작업에 첨부할 임의 메타데이터
```

## TaskGet (작업 조회)

```
설명: 작업 목록에서 ID로 작업을 조회합니다.

## 사용할 때
- 작업 시작 전 전체 설명과 맥락이 필요할 때
- 작업 의존성을 이해해야 할 때 (무엇을 차단하는지, 무엇이 차단하는지)
- 작업이 배정된 후 전체 요구사항을 얻기 위해

매개변수:
- taskId (필수, 문자열): 조회할 작업의 ID
```

## TaskList (작업 목록)

```
설명: 작업 목록의 모든 작업을 나열합니다.

## 사용할 때
- 작업 가능한 것 확인 (상태: 'pending', 소유자 없음, 차단되지 않음)
- 프로젝트의 전체 진행 상황 확인
- 차단되어 의존성 해결이 필요한 작업 찾기
- 팀원에게 작업 할당 전 사용 가능한 것 확인
- 작업 완료 후 새로 차단 해제된 작업 확인 또는 다음 작업 가져오기
- 여러 작업이 있을 때 ID 순서(가장 낮은 ID 먼저) 선호

매개변수: 없음
```

## TaskOutput (작업 출력)

```
설명: 실행 중이거나 완료된 작업(백그라운드 셸, 에이전트, 원격 세션)의 출력을 조회합니다.

매개변수:
- task_id (필수, 문자열): 출력을 가져올 작업 ID
- block (필수, 불리언, 기본값 true): 완료를 기다릴지 여부
- timeout (필수, 숫자, 기본값 30000, 최대 600000): 최대 대기 시간(ms)
```

## TaskStop (작업 중지)

```
설명: ID로 실행 중인 백그라운드 작업을 중지합니다.

매개변수:
- task_id (선택, 문자열): 중지할 백그라운드 작업의 ID
- shell_id (선택, 문자열): 더 이상 사용되지 않음: task_id를 대신 사용
```

## TaskUpdate (작업 업데이트)

```
설명: 작업 목록의 작업을 업데이트합니다.

## 사용할 때

**작업을 해결됨으로 표시:**
- 작업에 설명된 작업을 완료했을 때
- 작업이 더 이상 필요 없거나 대체되었을 때
- 중요: 할당된 작업을 완료하면 항상 해결됨으로 표시

- 완전히 달성했을 때만 작업을 완료로 표시
- 오류, 차단, 완료 불가 시 in_progress로 유지
- 절대 다음 경우에 완료로 표시하지 마세요:
  - 테스트 실패
  - 구현이 부분적
  - 해결되지 않은 오류 발생
  - 필요한 파일이나 의존성을 찾지 못함

## 상태 워크플로
상태 진행: `pending` → `in_progress` → `completed`
`deleted`로 작업을 영구 제거.

매개변수:
- taskId (필수, 문자열): 업데이트할 작업의 ID
- status (선택, 열거형): "pending", "in_progress", "completed", "deleted"
- subject (선택, 문자열): 새 제목
- description (선택, 문자열): 새 설명
- activeForm (선택, 문자열): in_progress 시 스피너에 표시될 현재진행형
- owner (선택, 문자열): 새 소유자
- metadata (선택, 객체): 병합할 메타데이터 키
- addBlocks (선택, 배열): 이 작업이 차단하는 작업 ID들
- addBlockedBy (선택, 배열): 이 작업을 차단하는 작업 ID들
```

## TeamCreate (팀 생성)

```
설명: 프로젝트에서 여러 에이전트의 작업을 조율하기 위해 새 팀을 생성합니다. 팀은 작업 목록과 1:1 대응합니다 (팀 = 작업목록).

## 사용할 때
다음 경우 사전적으로 사용:
- 사용자가 명시적으로 팀, 스웜, 에이전트 그룹 사용을 요청
- 사용자가 에이전트의 협력, 조율, 협업을 원한다고 언급
- 작업이 여러 에이전트의 병렬 작업이 유익할 정도로 복잡한 경우

## 팀원용 에이전트 유형 선택
Agent 도구로 팀원을 생성할 때, 에이전트에 필요한 도구에 따라 `subagent_type`을 선택하세요:
- 읽기 전용 에이전트 (Explore, Plan 등)는 파일 편집/쓰기 불가. 연구, 검색, 계획 작업만 배정.
- 전체 기능 에이전트 (general-purpose 등)는 파일 편집, 쓰기, bash 포함 모든 도구 접근 가능.

## 팀 워크플로
1. TeamCreate로 팀 생성
2. Task 도구로 작업 생성
3. Agent 도구로 팀원 생성 (team_name과 name 매개변수)
4. TaskUpdate로 유휴 팀원에게 작업 할당
5. 팀원이 배정된 작업을 수행하고 TaskUpdate로 완료 표시
6. 팀원은 턴 사이에 유휴 상태 — 정상적이고 예상된 것
7. SendMessage로 팀원에게 정상 종료 요청하여 팀 종료

## 자동 메시지 전달
중요: 팀원의 메시지는 자동으로 전달됩니다. 수신함을 수동으로 확인할 필요 없습니다.

## 팀원 유휴 상태
팀원은 매 턴 후 유휴 상태가 됩니다 — 완전히 정상이고 예상된 것입니다. 유휴 상태인 팀원에게 메시지를 보내면 깨어나서 정상적으로 처리합니다.

매개변수:
- team_name (필수, 문자열): 생성할 새 팀의 이름
- description (선택, 문자열): 팀 설명/목적
- agent_type (선택, 문자열): 팀 리더의 유형/역할
```

## TeamDelete (팀 삭제)

```
설명: 스웜 작업이 완료되면 팀과 작업 디렉터리를 제거합니다.

이 작업은:
- 팀 디렉터리 제거 (~/.claude/teams/{team-name}/)
- 작업 디렉터리 제거 (~/.claude/tasks/{team-name}/)
- 현재 세션에서 팀 컨텍스트 정리

중요: 팀에 아직 활성 멤버가 있으면 TeamDelete가 실패합니다. 먼저 팀원을 정상적으로 종료한 후 모든 팀원이 종료된 후에 TeamDelete를 호출하세요.

매개변수: 없음
```

## WebFetch (웹 가져오기)

```
설명: 중요: WebFetch는 인증되거나 비공개인 URL에서 실패합니다. 이 도구를 사용하기 전에 URL이 인증된 서비스(예: Google Docs, Confluence, Jira, GitHub)를 가리키는지 확인하세요. 그렇다면 인증된 접근을 제공하는 전문 MCP 도구를 찾으세요.

- 지정된 URL에서 콘텐츠를 가져오고 AI 모델을 사용하여 처리합니다
- URL과 프롬프트를 입력으로 받습니다
- URL 콘텐츠를 가져와 HTML을 마크다운으로 변환
- 작고 빠른 모델을 사용하여 프롬프트로 콘텐츠를 처리
- 콘텐츠에 대한 모델의 응답을 반환

사용 참고:
- MCP 제공 웹 가져오기 도구가 있으면 이 도구 대신 그것을 선호하세요
- URL은 완전한 유효 URL이어야 합니다
- HTTP URL은 자동으로 HTTPS로 업그레이드됩니다
- 더 빠른 응답을 위한 15분 자기 정리 캐시 포함
- GitHub URL의 경우 대신 Bash를 통해 gh CLI를 사용하세요

매개변수:
- url (필수, 문자열, URI 형식): 콘텐츠를 가져올 URL
- prompt (필수, 문자열): 가져온 콘텐츠에 대해 실행할 프롬프트
```

## WebSearch (웹 검색)

```
설명: Claude가 웹을 검색하고 결과를 사용하여 응답에 정보를 제공할 수 있습니다.

- 현재 이벤트와 최신 데이터에 대한 최신 정보를 제공
- 마크다운 하이퍼링크를 포함한 검색 결과 블록으로 형식화된 정보 반환
- Claude의 지식 컷오프를 넘어서는 정보에 접근할 때 사용

필수 요구사항:
- 사용자의 질문에 답한 후, 응답 끝에 "Sources:" 섹션을 반드시 포함해야 합니다
- Sources 섹션에 검색 결과의 모든 관련 URL을 마크다운 하이퍼링크로 나열: [제목](URL)
- 이것은 필수입니다 - 응답에 소스 포함을 절대 건너뛰지 마세요

중요 - 검색 쿼리에 올바른 연도를 사용하세요:
- 현재 월은 2026년 3월입니다. 최신 정보, 문서, 현재 이벤트를 검색할 때 이 연도를 반드시 사용하세요.

매개변수:
- query (필수, 문자열, 최소 2자): 사용할 검색 쿼리
- allowed_domains (선택, 배열): 이 도메인의 검색 결과만 포함
- blocked_domains (선택, 배열): 이 도메인의 검색 결과를 절대 포함하지 않음
```
